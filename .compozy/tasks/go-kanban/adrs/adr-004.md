# ADR-004: Layered Go Backend with Server-Confirmed UI Updates

## Context

The backend needs enough structure to keep business rules maintainable, especially around board, column, and card ordering. The user also chose to avoid optimistic UI updates in the MVP to reduce frontend state reconciliation complexity.

## Decision

We will structure the Go backend using three layers:

- Handlers for HTTP and request-response mapping
- Services for business rules and orchestration
- Repositories for persistence

The frontend will only reflect card and column changes after successful API responses. Drag-and-drop interactions will therefore be server-confirmed rather than optimistic.

## Alternatives Considered

### Alternative 1: Handler-only backend

- **Description**: Place validation, business rules, and persistence logic directly in HTTP handlers.
- **Pros**: Less boilerplate and faster initial coding.
- **Cons**: Harder to test, harder to migrate storage, and likely to entangle transport with domain logic.
- **Why rejected**: Ordering rules and template creation justify a clearer separation.

### Alternative 2: Hexagonal architecture from the start

- **Description**: Adopt ports and adapters with stricter boundaries across the entire backend.
- **Pros**: Strong architectural purity and long-term flexibility.
- **Cons**: More upfront abstraction and slower MVP delivery.
- **Why rejected**: The MVP does not yet need that level of indirection.

## Consequences

### Positive

- Centralized domain rules in service code
- Cleaner unit and integration testing boundaries
- Easier future storage migration

### Negative

- More files and interfaces than a minimal handler-only design
- Slightly slower initial implementation

### Risks

- The UI may feel less responsive without optimistic updates
- Mitigation: keep move operations fast and show clear loading state during persistence

## Implementation Notes

- Domain errors should be defined once and mapped by handlers
- Ordering operations should live in services and use transactional repository methods
