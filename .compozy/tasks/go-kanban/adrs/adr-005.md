# ADR-005: Docker and Docker Compose for Reproducible Local and Agent Execution

## Context

The project will be developed and tested across local environments and AI-assisted runtimes such as Codex and Claude. Host toolchains can drift, and some environments are easier to standardize through containers than through direct host setup.

## Decision

We will provide Dockerfiles for the frontend and backend plus a Docker Compose configuration that supports:

- Local multi-service startup
- Stable ports and environment variables
- Mounted source code for development workflows
- Reproducible test execution in containerized environments

Containerized execution will be the default verification path in agent-driven environments.

## Alternatives Considered

### Alternative 1: Host-only setup instructions

- **Description**: Rely on local Go and Node installations with documented versions.
- **Pros**: Less container maintenance and often faster host-native iteration.
- **Cons**: Higher risk of environment drift and harder reproducibility across agent runtimes.
- **Why rejected**: The project explicitly wants reliable execution across different automated environments.

### Alternative 2: Ad hoc scripts per runtime

- **Description**: Create separate scripts for Claude, Codex, CI, and local development.
- **Pros**: Can be tailored to each runtime.
- **Cons**: Fragments the workflow and creates maintenance duplication.
- **Why rejected**: Compose offers a cleaner shared contract for local and automated execution.

## Consequences

### Positive

- Reproducible development and test environments
- Lower onboarding friction when Docker is available
- Better consistency across local and agent execution

### Negative

- Additional maintenance for Dockerfiles and compose configuration
- Possible performance penalties on some hosts with bind mounts

### Risks

- Poorly configured volumes or startup commands can make development slower
- Mitigation: keep images minimal, separate dev and test targets, and use focused mounts

## Implementation Notes

- Frontend and backend should each have a dedicated Dockerfile
- Compose should include a straightforward full-stack startup path and a test path
