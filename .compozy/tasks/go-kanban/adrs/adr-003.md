# ADR-003: SQLite for MVP Persistence with Future PostgreSQL Migration Path

## Context

The PRD calls for durable persistence and originally mentions PostgreSQL in the MVP scope. The user chose SQLite for the first implementation to reduce operational overhead while preserving the option to migrate to PostgreSQL later. The MVP is explicitly single-user, which lowers the concurrency burden.

## Decision

We will use SQLite as the persistence engine for the MVP and isolate data access behind repository interfaces so the service layer remains database-agnostic enough for a future PostgreSQL migration.

The backend will store boards, columns, cards, and ordering metadata in SQLite and enable durability features appropriate for local usage.

## Alternatives Considered

### Alternative 1: PostgreSQL from day one

- **Description**: Start directly with PostgreSQL and keep the original PRD persistence target unchanged.
- **Pros**: Better concurrency characteristics, smoother path to future scaling, production-like setup.
- **Cons**: Higher setup and operational overhead for the first usable MVP.
- **Why rejected**: The MVP prioritizes speed and simplicity for a single-user workflow.

### Alternative 2: In-memory persistence for the MVP

- **Description**: Use in-memory data structures and defer persistence until later.
- **Pros**: Fastest possible implementation and minimal storage complexity.
- **Cons**: Violates the core requirement for durable data and persistence across restarts.
- **Why rejected**: It fails an explicit MVP requirement.

## Consequences

### Positive

- Minimal local setup for development and testing
- Durable persistence without requiring a separate database server
- Good fit for single-user local-first usage

### Negative

- Weaker concurrent write characteristics than PostgreSQL
- Future migration will require schema and query adaptation

### Risks

- SQLite locking behavior can surface with concurrent writes from multiple tabs
- Mitigation: enable WAL mode, keep write transactions short, and centralize ordering writes

## Implementation Notes

- Repositories should avoid storage-specific leakage into services
- Migration files should remain explicit to simplify eventual PostgreSQL translation
