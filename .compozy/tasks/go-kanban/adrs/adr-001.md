# ADR-001: MVP Lean for Go Kanban

## Context

The team needs to build a personal Kanban board system with Go backend and Next.js frontend. Multiple approaches were considered for the MVP scope, balancing delivery speed with feature completeness.

## Decision

We will pursue the **MVP Lean** approach:

- Personal board with pre-defined templates (Basic Kanban, Bug Tracker, Content Pipeline)
- CRUD for boards and customizable columns
- Cards with title, description, and drag-and-drop between columns
- PostgreSQL persistence
- Clean, functional interface without visual polish overhead

## Alternatives Considered

### Approach B: Focused on UX
- Same scope as A + smooth drag-and-drop animations
- Optimistic UI feedback
- Light/dark themes
- *Trade-off: More time on frontend polish, delays core functionality delivery*

### Approach C: Data-First
- MVP A + card movement history
- Basic metrics (cards per column, cards per day)
- Board export as JSON
- *Trade-off: More initial complexity, but prepares for analytics earlier*

## Consequences

### Positive
- Faster time to first working version
- Clear, minimal scope reduces risk of scope creep
- Solid foundation for iterative feature additions
- Templates provide immediate value without complex customization UI

### Negative
- Interface will be functional but not polished initially
- No analytics or history tracking in MVP
- UX improvements will need to be added in subsequent iterations

### Risks
- Users may find the interface too bare without visual feedback
- Drag-and-drop without optimistic updates may feel sluggish
- Mitigation: Plan UX improvements as first post-MVP tasks
