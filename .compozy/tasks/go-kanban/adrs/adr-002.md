# ADR-002: Separate Next.js Frontend and Go REST Backend

## Context

The MVP must use Go for backend responsibilities and Next.js for the frontend experience. The project needs clear boundaries between UI concerns and domain or persistence logic while keeping the first implementation straightforward enough to deliver quickly.

## Decision

We will implement the system as two separate applications:

- A Next.js frontend responsible for rendering the user interface and managing user interactions
- A Go backend responsible for exposing REST APIs, applying business rules, and persisting data

The frontend will communicate with the backend through explicit HTTP JSON contracts.

## Alternatives Considered

### Alternative 1: Next.js BFF with minimal Go service

- **Description**: Use Next.js as both UI and backend-for-frontend, with Go handling only selected domain operations.
- **Pros**: Fewer moving parts in the frontend stack, easier page-to-data integration.
- **Cons**: Splits backend responsibility across two runtimes and weakens domain boundaries.
- **Why rejected**: The MVP benefits more from keeping all domain and persistence logic in Go behind a single API surface.

### Alternative 2: Monolithic Go server with embedded frontend delivery

- **Description**: Serve the entire application from a single Go runtime, possibly bundling frontend assets.
- **Pros**: Fewer deployable services and simpler runtime topology.
- **Cons**: Reduces frontend flexibility and makes the Next.js choice less useful.
- **Why rejected**: It conflicts with the requirement to use Next.js as a first-class frontend application.

## Consequences

### Positive

- Clear separation between UI and domain logic
- Explicit API contracts that simplify testing and future replacement
- Better alignment with the chosen frontend and backend technologies

### Negative

- Requires CORS or proxy configuration during development
- Introduces two application runtimes instead of one

### Risks

- API contracts may drift from frontend expectations
- Mitigation: define stable DTOs and cover critical flows with integration tests

## Implementation Notes

- The frontend will call the backend through REST endpoints under `/api`
- The backend will own all board, column, and card rules
